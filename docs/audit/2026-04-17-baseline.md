## 1. RESUMEN EJECUTIVO (10 líneas)

`aionseg.co` **no es una landing estática**: es una plataforma VMS empresarial en producción con 31 procesos PM2, 162 tablas PostgreSQL, 129 streams de video go2rtc, 42 endpoints PJSIP Asterisk, IAM JWT local, y 11 contenedores Docker (observabilidad + MinIO recién desplegado).

El stack objetivo del contrato (Frigate, Leosac, Keycloak, Graylog, OpenALPR, Grafana, Node-RED, Mosquitto, MinIO, PostgreSQL) coincide **parcialmente** con lo ya instalado. **Mosquitto**, **PostgreSQL 16**, **Grafana + Prometheus + Alertmanager**, **MinIO** y **Nginx con TLS válido 74 días** ya están corriendo. **Frigate, Leosac, Keycloak, Graylog, OpenALPR no existen todavía**.

El VPS tiene margen holgado: 25 GB RAM libres, 93 GB en disco secundario `/data` casi vacío, 8 vCPU al 10% de uso. **No es necesario edge-split salvo para Frigate con >30 cámaras simultáneas**. El modo híbrido (VPN WireGuard + 2 servidores locales del cliente) ya está preparado pero sin peers conectados.

**Principales riesgos:** Mosquitto con `allow_anonymous true` y sin ACL; Redis con `NOAUTH Authentication required` (configuración inconsistente); ausencia de Keycloak — el ecosistema hoy depende de `@fastify/jwt` HS256 que no soporta SSO multi-servicio. El modelo de eventos canónico del contrato **no existe** — hay tablas por-dominio (`events`, `intercom_calls`, etc.) pero no un bus MQTT con schema v1 validado.

### 4.3 Conclusión capacidad

**El VPS SOPORTA el stack objetivo completo con estas condiciones:**

1. **Frigate con ≤8 cámaras de baja resolución (720p) sin hwaccel en VPS** → OK.
2. **Frigate con >8 cámaras, ≥1080p, o detección IA pesada** → **EDGE OBLIGATORIO** en servidor local 32 GB i5.
3. **Graylog + Elasticsearch** → Sustituir por **Loki + Promtail** (4x menor footprint) para mantener margen. Evaluar Graylog solo si el cliente requiere explícitamente su UI.
4. **PostgreSQL** → ya optimizado (`shared_buffers 8GB`, `work_mem 64MB`), tiene capacidad para `aion_core` + `keycloak` + `aionseg_prod`.
5. Reservar **15-20% RAM (~5 GB)** sin asignar como headroom para picos (eventos masivos, backups, n8n workflows, compilaciones).

## 6. GARANTÍAS PROPUESTAS (VALIDACIÓN AUTOMATIZADA)

Cada fase tendrá un `scripts/gates/phase-N.sh` que valida:

### 6.1 Garantías de integridad

- **Healthchecks**: todos los nuevos contenedores exponen `/health` (liveness) y `/ready` (readiness).
- **Tests con cobertura ≥80%** antes de cierre de fase (pytest para Python, vitest para TS).
- **docker compose up -d** desde cero tras `docker compose down -v` debe dejar todos los healthchecks verdes en <5 min.

### 6.2 Garantías de seguridad

- **Grep pre-commit** regex `(password|api_key|secret|token)=.{8,}` bloquea PRs con secretos.
- **Nmap desde WAN** no debe encontrar puertos nuevos fuera de 22/80/443 sin ADR justificatorio.
- **testssl.sh aionseg.co** debe devolver grado A o superior.
- **SBOM** generado por fase (`syft` o `trivy fs`).

### 6.3 Garantías de observabilidad

- **Prometheus scrape targets** 100% UP al cierre de fase (verificado vía API `prometheus/api/v1/targets`).
- **Alertmanager** tiene al menos 1 alerta crítica por servicio nuevo (p.ej. `service_down_5m`).
- **Logs JSON** con `correlation_id` por evento verificados con `grep -E '\"correlation_id\":\"[0-9a-f-]{36}\"'`.

### 6.4 Garantías de backup/recuperación

- **Postgres dump diario** a `/data/backups/postgres/YYYY-MM-DD.dump` (cron ya tiene infra).
- **MinIO versioning + lifecycle** en buckets críticos (`aion-backups`, `frigate-clips`).
- **Restore drill** trimestral documentado en `docs/runbooks/restore-drill.md`.

### 6.5 Garantías de rollback

- **Cada fase** crea un tag git `phase-N-pre` antes del PR y `phase-N-complete` al cierre.
- **Snapshot EBS** vía AWS Console o CLI antes de cualquier Fase que toque infra (disponible en `docs/runbooks/ebs-snapshot.md`).
- **docker compose fragments** son **aditivos** — nunca se modifica/elimina uno existente sin ADR.
